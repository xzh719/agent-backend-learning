1Tool Calling 深度解析

Tool Calling 是 Agent 的「手」——让 LLM 从纯文本生成器变成能操作外部世界的执行者。掌握它，就掌握了 Agent 的核心能力。

1.1 Tool Calling 完整流程

用户: "北京今天天气如何？" │ ▼ ┌─ Agent Loop ──────────────────────────────────────┐ │ Step 1: LLM 推理 → 决定调 tool │ │ messages = [user: "北京今天天气如何？"] │ │ tools = [{name:"get_weather", params:{city}}] │ │ │ │ Step 2: LLM 返回 tool_call (不是文本回复!) │ │ tool_calls = [{ │ │ id: "call_abc123", │ │ function: {name: "get_weather", arguments: {city: "北京"}}│ │ }] │ │ │ │ Step 3: Agent 后端执行工具 │ │ result = await get_weather("北京") │ │ → "北京今天晴，22°C，湿度 45%" │ │ │ │ Step 4: 把结果拼回 messages，再次调 LLM │ │ messages.append(tool_result) │ │ → LLM 生成最终回复: "北京今天天气晴朗，22°C..." │ └────────────────────────────────────────────────────┘

1.2 工具定义规范

# OpenAI Function Calling 的标准工具定义
TOOLS = [
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "获取指定城市的当前天气信息。返回温度、湿度、天气状况。",
            "parameters": {
                "type": "object",
                "properties": {
                    "city": {
                        "type": "string",
                        "description": "城市名称，如 '北京'、'上海'",
                    },
                    "unit": {
                        "type": "string",
                        "enum": ["celsius", "fahrenheit"],
                        "description": "温度单位",
                    }
                },
                "required": ["city"],
            }
        }
    }
]

# ★ 关键原则 ★
# 1. description 是给 LLM 看的「使用说明书」，越清晰 LLM 越不容易误用
# 2. enum 优于 string —— 约束越紧，模型越难出错
# 3. required 字段要精确 —— LLM 有时会省略非 required 参数
# 4. 工具名要有动词 —— "get_weather" 而非 "weather"

1.3 tool_choice 参数详解

值含义何时用
"auto"LLM 自己决定是否调工具默认，大多数情况
"none"禁止调工具只要纯文本回复时
"required"必须调一个工具结构化输出 / 强制行动
{"type":"function","function":{"name":"X"}}指定必须调 X 工具采集特定结构化数据
tool_choice="required" 的陷阱： 某些模型（DeepSeek、Groq）在 tool_choice="required" 但不需要调工具时，会「发明」工具调用（幻觉）。只在确实需要强制行动时才用 required。

1.5 工具执行的安全边界

class ToolExecutor:
    """安全执行 LLM 返回的 tool calls"""

    ALLOWED_TOOLS = {"get_weather", "search", "read_file"}  # 白名单!
    MAX_TIMEOUT = 30   # 秒
    MAX_RESULT_SIZE = 50000  # 字节

    async def execute(self, tool_calls: list) -> list:
        results = []
        for tc in tool_calls:
            func_name = tc.function.name
            if func_name not in self.ALLOWED_TOOLS:
                results.append({"error": f"Tool '{func_name}' not allowed"})
                continue

            try:
                args = json.loads(tc.function.arguments)
                async with asyncio.timeout(self.MAX_TIMEOUT):
                    result = await self.registry[func_name](**args)
                    # 截断过大的结果
                    result_str = str(result)[:self.MAX_RESULT_SIZE]
                    results.append({"tool_call_id": tc.id, "result": result_str})
            except asyncio.TimeoutError:
                results.append({"error": "Tool timeout"})
            except Exception as e:
                results.append({"error": str(e)})
        return results

2ReAct 循环引擎

ReAct (Reasoning + Acting) 是 Agent 的标准运行模式。LLM 在「思考→行动→观察」的循环中自主推进任务，直到得出结论。

2.1 ReAct 状态机

┌──────────────────────┐ │ START │ │ user sends message │ └──────────┬───────────┘ │ ▼ ┌──────────────────────┐ ┌───→│ LLM REASONING │ │ │ (Call Chat API) │ │ └──────────┬───────────┘ │ │ │ ┌────────┴────────┐ │ │ │ │ finish_reason finish_reason │ = "stop" = "tool_calls" │ │ │ │ ▼ ▼ │ ┌────────┐ ┌──────────────┐ │ │ DONE │ │ EXECUTE TOOLS│ │ └────────┘ └──────┬───────┘ │ │ │ ▼ │ ┌──────────────────┐ │ │ ADD TOOL RESULTS │ │ │ to messages │ │ └────────┬─────────┘ │ │ └───────────────────────┘

2.2 核心 Agent Loop 实现

class AgentLoop:
    """一个完整的 ReAct Agent Loop"""

    def __init__(self, llm, tools: dict, max_steps: int = 15):
        self.llm = llm          # BaseLLMProvider 实例
        self.tools = tools      # {name: callable}
        self.max_steps = max_steps
        self.messages = []

    async def run(self, user_input: str, system_prompt: str = "") -> str:
        self.messages = []
        if system_prompt:
            self.messages.append(LLMMessage(role="system", content=system_prompt))
        self.messages.append(LLMMessage(role="user", content=user_input))

        for step in range(self.max_steps):
            self.emit("step", step)

            # 1. 调 LLM
            response = await self.llm.chat(
                messages=self.messages,
                tools=self._tool_definitions(),
                temperature=0.3,
            )

            # 2. 判断结果类型
            if response.tool_calls:
                self.emit("tool_calls", response.tool_calls)

                # 3. 执行工具
                tool_results = await self._execute_tools(response.tool_calls)
                self.emit("tool_results", tool_results)

                # 4. 把 tool_call + tool_result 拼回消息
                self.messages.append(LLMMessage(
                    role="assistant",
                    content="",
                    tool_calls=response.tool_calls,
                ))
                for tr in tool_results:
                    self.messages.append(LLMMessage(
                        role="tool",
                        content=tr["result"],
                        tool_call_id=tr["tool_call_id"],
                    ))

            elif response.content:
                # 纯文本回复 → Agent 结束
                self.emit("done", response.content)
                return response.content

        # 超过最大步数
        raise RuntimeError(f"Agent exceeded max steps ({self.max_steps})")

    def emit(self, event_type, data=None):
        """事件钩子——前端 WebSocket/SSE 用这个推送进度"""
        print(f"[{event_type}]", data)

2.3 关键设计决策

决策点选项 A选项 B建议
多轮后是否清空历史?保留全部 → context 越来越长只保留当前循环 → 丢失上下文滑动窗口裁剪（阶段 2 技术）

2.4 中断与恢复

class InterruptibleAgent(AgentLoop):
    """用户可随时中断的 Agent"""

    def __init__(self, *args, **kwargs):
        super().__init__(*args, **kwargs)
        self._cancelled = False

    def cancel(self):
        self._cancelled = True

    async def run(self, user_input, system_prompt=""):
        self._cancelled = False
        for step in range(self.max_steps):
            if self._cancelled:
                return "[CANCELLED BY USER]"
            # ... rest of the loop ...

    def checkpoint(self) -> dict:
        """保存当前状态 → 可以恢复"""
        return {
            "messages": self.messages,
            "step": self.current_step,
        }

    def restore(self, checkpoint: dict):
        self.messages = checkpoint["messages"]
        self.current_step = checkpoint["step"]
Agent Loop 的「秘密」： 整个 ReAct 循环没有任何 AI 魔法——就是普通代码：while 循环里调 LLM + 执行工具 + 拼消息。难点不在代码，而在于 ① 工具定义清晰度 ② 错误恢复策略 ③ context 管理 ④ 中断机制。

3任务规划与分解

简单问题一问一答，复杂任务需要先规划再执行。Plan → Execute → Verify 三段式是生产级 Agent 的标准范式。

3.1 两种规划策略

Plan-and-Execute (计划先行): User Input → Plan → [Task1, Task2, Task3] → Execute each → Verify → Done ↑ 适合任务明确、步骤可枚举的场景 ↓ 缺点：计划错了就全错了，没有动态调整空间 ReAct (边想边做): User Input → Think → Act → Observe → Think → Act → Observe → ... → Done ↑ 适合开放性问题、需要探索的场景 ↓ 缺点：没有全局视角，可能走弯路 ★ 最佳实践: Plan + ReAct 混合 —— 先出计划（Plan），执行时用 ReAct（允许动态调整）

3.2 Plan Generator —— 用 LLM 生成执行计划

async def generate_plan(task: str, llm) -> list[dict]:
    """让 LLM 把复杂任务拆解成可执行的步骤列表"""
    plan_response = await llm.chat(
        messages=[{"role": "user", "content": f"""Task: {task}

Break this task into steps. For each step, specify:
1. What to do (action)
2. What tool to use (tool) — one of: search, read_file, write_file, execute_python
3. Expected output (expected)
4. Dependencies (depends_on) — which step IDs must complete first

Return as JSON array:
[{{"id": 1, "action": "...", "tool": "...", "expected": "...", "depends_on": []}}]
"""}],
        temperature=0.2,
        response_format={"type": "json_object"},
    )
    return json.loads(plan_response.content)["steps"]

3.3 Plan Executor —— 按 DAG 执行步骤

class PlanExecutor:
    """根据 DAG 依赖关系执行 Plan"""

    def __init__(self, agent: AgentLoop):
        self.agent = agent

    async def execute(self, steps: list[dict]) -> list[dict]:
        completed = {}  # step_id → result
        pending = steps.copy()

        while pending:
            # 找出所有依赖已满足的步骤
            ready = [
                s for s in pending
                if all(dep in completed for dep in s.get("depends_on", []))
            ]
            if not ready:
                raise RuntimeError("Deadlock: no ready steps but tasks remain")

            # 并行执行所有 ready 步骤
            tasks = [self.agent.run(s["action"]) for s in ready]
            results = await asyncio.gather(*tasks, return_exceptions=True)

            for step, result in zip(ready, results):
                if isinstance(result, Exception):
                    completed[step["id"]] = {"error": str(result)}
                else:
                    completed[step["id"]] = {"result": result}
                pending.remove(step)

        return completed

3.4 Reflection —— 自我反思修正

async def reflect_and_improve(llm, plan_result: str, goal: str) -> str:
    """执行完后自我检查：是否达到目标？如果没达到，自动修正"""
    check = await llm.chat(messages=[
        {"role": "user", "content": f"""Goal: {goal}

Plan execution result:
{plan_result}

Evaluate: Did we achieve the goal?
- If YES: Reply "PASS"
- If NO: Reply "NEEDS_FIX: [what's wrong]"

Then provide corrected output if needed.
"""}
    ], temperature=0)

    if "NEEDS_FIX" in check.content:
        # 发现问题，再跑一轮修正
        fix = await llm.chat(messages=[
            {"role": "user", "content": f"Original result:\n{plan_result}

Problem: {check.content}

Please fix it."}
        ])
        return fix.content
    return plan_result
Plan + ReAct 混合模式总结：
① LLM Plan：把大任务拆成 3-8 个步骤 → ② DAG Executor：按依赖并行执行 → ③ 每个步骤内部用 ReAct：允许 LLM 在步骤内灵活调整 → ④ Reflection：执行完后自检，必要时自动修正。
这就是大多数生产级 Agent 框架（LangGraph、AutoGPT、CrewAI）的底层架构。

4状态机与流程控制

Agent 不是简单的 while 循环。生产级 Agent 需要显式的状态机来管理生命周期、处理异常、实现复杂的多阶段流程。

4.1 Agent 生命周期状态机

┌─────────┐ │ IDLE │ ← 等待用户输入 └────┬────┘ │ user_message ▼ ┌─────────┐ ┌────│THINKING │────┐ │ └─────────┘ │ │ tool_calls │ text_response ▼ ▼ ┌─────────┐ ┌─────────┐ │EXECUTING│ │FINISHED │ └────┬────┘ └─────────┘ │ ▼ ┌─────────┐ ┌───│OBSERVING│───┐ │ └─────────┘ │ │ continue │ done │ ▼ │ ┌─────────┐ └───────────→│THINKING │ (回到思考) ┌────────────│ (or) │ │ └─────────┘ │ error ▼ ┌─────────┐ │ ERROR │──→ IDLE (用户决定下一步) └─────────┘

4.2 用 Python 实现状态机

from enum import Enum, auto
from typing import Callable

class AgentState(Enum):
    IDLE = auto()
    THINKING = auto()
    EXECUTING = auto()
    OBSERVING = auto()
    FINISHED = auto()
    ERROR = auto()
    CANCELLED = auto()

class StateMachineAgent:
    """显式状态机的 Agent"""

    TRANSITIONS = {
        AgentState.IDLE:      [AgentState.THINKING, AgentState.CANCELLED],
        AgentState.THINKING:  [AgentState.EXECUTING, AgentState.FINISHED, AgentState.ERROR],
        AgentState.EXECUTING: [AgentState.OBSERVING, AgentState.ERROR],
        AgentState.OBSERVING: [AgentState.THINKING, AgentState.FINISHED, AgentState.ERROR],
        AgentState.ERROR:     [AgentState.IDLE, AgentState.THINKING],
    }

    def __init__(self, max_loops=15):
        self.state = AgentState.IDLE
        self.loop_count = 0
        self.max_loops = max_loops
        self.handlers: dict[AgentState, Callable] = {}
        self.history: list[tuple[AgentState, AgentState]] = []

    def on(self, state: AgentState):
        """装饰器——注册状态处理函数"""
        def decorator(fn):
            self.handlers[state] = fn
            return fn
        return decorator

    def transition(self, new_state: AgentState):
        if new_state not in self.TRANSITIONS.get(self.state, []):
            raise ValueError(f"Invalid transition: {self.state} → {new_state}")
        self.history.append((self.state, new_state))
        self.state = new_state

    async def run(self):
        while self.state != AgentState.FINISHED and self.loop_count < self.max_loops:
            handler = self.handlers[self.state]
            await handler()
            self.loop_count += 1

agent = StateMachineAgent()

@agent.on(AgentState.THINKING)
async def handle_thinking():
    response = await llm.chat(...)
    if response.tool_calls:
        agent.transition(AgentState.EXECUTING)
    else:
        agent.transition(AgentState.FINISHED)

@agent.on(AgentState.ERROR)
async def handle_error():
    # 记录错误、尝试恢复、或者返回友好提示
    if retry_count < 3:
        agent.transition(AgentState.THINKING)
    else:
        agent.transition(AgentState.IDLE)

5MCP 协议集成

MCP (Model Context Protocol) 是 Anthropic 提出的 Agent-工具通信标准。它让你的 Agent 可以通过统一协议连接任何 MCP Server，而不是每个工具都写一次对接代码。

5.3 MCP Client 实现

import asyncio, json

class MCPClient:
    """通过 stdio 连接 MCP Server"""

    def __init__(self, server_command: list[str]):
        self.cmd = server_command  # e.g., ["npx", "@modelcontextprotocol/server-github"]
        self.process = None
        self.request_id = 0

    async def connect(self):
        self.process = await asyncio.create_subprocess_exec(
            *self.cmd,
            stdin=asyncio.subprocess.PIPE,
            stdout=asyncio.subprocess.PIPE,
        )

    async def _rpc(self, method: str, params: dict = {}) -> dict:
        self.request_id += 1
        request = json.dumps({
            "jsonrpc": "2.0",
            "id": self.request_id,
            "method": method,
            "params": params,
        })
        self.process.stdin.write((request + "\n").encode())
        await self.process.stdin.drain()

        raw = await self.process.stdout.readline()
        return json.loads(raw)

    async def list_tools(self) -> list:
        resp = await self._rpc("tools/list")
        return resp["result"]["tools"]

    async def call_tool(self, name: str, args: dict) -> str:
        resp = await self._rpc("tools/call", {"name": name, "arguments": args})
        return resp["result"]

        all_tools.extend(mcp_to_openai_tool(t) for t in mcp_tools)
    return all_tools

6.2 Orchestrator 实现

class Orchestrator:
    """协调多个 Worker Agent 完成复杂任务"""

    def __init__(self, llm, workers: dict[str, AgentLoop]):
        self.llm = llm
        self.workers = workers  # {"coder": Agent, "reviewer": Agent, "searcher": Agent}

    async def delegate(self, task: str) -> str:
        # Step 1: Orchestrator 分析任务，决定分工
        plan = await self.llm.chat(messages=[{
            "role": "user",
            "content": f"""Available workers: {list(self.workers.keys())}

Task: {task}

Plan which worker should do what, in what order.
Return JSON: [{{"step": 1, "worker": "...", "task": "..."}}]
"""
        }], response_format={"type": "json_object"})
        steps = json.loads(plan.content)["steps"]

        # Step 2: 顺序执行（有依赖）或并行执行（无依赖）
        context = {}
        for step in steps:
            worker = self.workers[step["worker"]]
            task_with_context = f"{step['task']}

Context from previous steps:
{json.dumps(context, indent=2, ensure_ascii=False)}"
            result = await worker.run(task_with_context)
            context[step["worker"]] = result

        # Step 3: Orchestrator 汇总所有 Worker 结果
        summary = await self.llm.chat(messages=[{
            "role": "user",
            "content": f"Original task: {task}

Worker outputs:
{json.dumps(context, indent=2, ensure_ascii=False)}

Synthesize a final response."
        }])
        return summary.content

6.4 多 Agent 通信协议

from dataclasses import dataclass
import asyncio

@dataclass
class AgentMessage:
    from_agent: str
    to_agent: str
    type: str    # "task" | "result" | "query" | "cancel"
    content: str

class AgentBus:
    """Agent 间消息总线（类似微服务的 Message Queue）"""

    def __init__(self):
        self.queues: dict[str, asyncio.Queue] = {}
        self.broadcast: list = []

    def register(self, agent_name: str):
        self.queues[agent_name] = asyncio.Queue()

    async def send(self, msg: AgentMessage):
        if msg.to_agent in self.queues:
            await self.queues[msg.to_agent].put(msg)
        else:
            self.broadcast.append(msg)  # 目标不存在 → 广播

    async def receive(self, agent_name: str) -> AgentMessage:
        return await self.queues[agent_name].get()
多 Agent 的反模式： ① 过度拆分 → 3 个 Agent 就够的活用了 10 个，协调开销吃掉所有收益
② Agent 互相甩锅 → "不是我的问题，你去找 Agent B"的死循环
③ 信息丢失 → Orchestrator 转发时省略关键细节（Chinese whispers 效应）
④ 成本爆炸 → 每个 Agent 都在调 LLM，5 个 Agent 一轮就是 5 次 API 调用
原则：先做好单 Agent，再考虑多 Agent。
