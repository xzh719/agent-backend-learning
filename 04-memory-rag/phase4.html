1Embedding 向量化

Embedding 是 RAG 和 Agent 记忆的基石。把文本变成高维向量，语义相近的文本向量也相近。

1.1 什么是 Embedding？

文本: "猫是一种可爱的宠物" → Embedding Model (e.g., text-embedding-3-small) → 向量: [0.023, -0.451, 0.782, ..., 0.112] ← 1536 维 文本: "狗是人类的好朋友" → Embedding Model → 向量: [0.019, -0.443, 0.791, ..., 0.108] cosine_similarity(猫, 狗) = 0.87 ← 高相似度 ✓ cosine_similarity(猫, "TCP 协议") = 0.12 ← 低相似度 ✓

1.2 主流 Embedding 模型速查

模型维度最大 Token成本 ($/1M tokens)特点
text-embedding-3-small512/15368191$0.02性价比王，dimension 可变
text-embedding-3-large256/30728191$0.13精度最高
text-embedding-ada-00215368191$0.10旧版，不推荐

1.4 本地 Embedding 方案 (sentence-transformers)

from sentence_transformers import SentenceTransformer

# 加载模型（首次会下载 ~200MB-1GB）
model = SentenceTransformer("BAAI/bge-large-zh-v1.5")

texts = ["Python 是一门编程语言", "今天天气很好"]

# 生成 embedding

    texts,
    normalize_embeddings=True,   # L2 归一化 → 余弦相似度 = 内积

    batch_size=32,

from sklearn.metrics.pairwise import cosine_similarity
sim = cosine_similarity([embeddings[0]], [embeddings[1]])[0][0]
print(f"相似度: {sim:.4f}")

1.5 Embedding 质量的关键因素

因素影响最佳实践
文本切分 (Chunking)太大→语义混杂；太小→缺少上下文256-1024 tokens，50-100 token 重叠
文本预处理格式垃圾 → 噪音向量去掉 HTML/Markdown 标记、多余空格
查询改写短查询 → 信息不足HyDE 技术：用 LLM 扩展查询再 embedding
维度选择越高越精确但越慢512 对大多数场景够用
多语言跨语言检索可能失败用多语言模型（bge-m3, Cohere v3）

2向量数据库

向量数据库是 Agent 的「外部大脑」——存储海量文档的向量，毫秒级检索最相关的内容。

2.1 向量数据库选型

数据库部署性能适合场景特点
ChromaDB嵌入式/Python中等开发/原型零配置，pip install 即用
Qdrant独立服务/Docker高生产环境Rust 编写，过滤强，有云服务
Milvus分布式集群极高十亿级向量企业级，K8s 部署
Weaviate独立服务高混合检索自带向量化 + 关键词混合
PineconeSaaS高不想管运维全托管，但贵
PostgreSQL + pgvector已有 PG中等已有 PG 的团队无额外基础设施
FAISS库极高离线/嵌入式Meta 出品，无持久化

2.4 索引类型选择

索引延迟召回率内存适合
HNSW (分层可导航小世界图)低~95%高通用，速度快
IVF (倒排文件)中~90-98%中大规模 (100万+)
Flat (暴力搜索)高 (线性)100%低小数据集 (<1万)
经验法则：数据量 < 10 万条 → Flat；10 万 ~ 100 万 → HNSW；> 100 万 → IVF/HNSW + 分区。

3RAG 全链路

RAG = Retrieval-Augmented Generation（检索增强生成）。核心思想：先把相关文档检索出来，塞进 LLM 的 context，让 LLM 基于这些文档回答。

3.1 RAG Pipeline 全貌

┌────────────────────────────────────────────────────────────┐ │ INDEXING (离线 / 文档导入时) │ │ │ │ 文档 ──→ Text Splitter ──→ Embedder ──→ Vector DB │ │ PDF Chunk 1 [vec1] 存储 chunk + vector │ │ HTML Chunk 2 [vec2] │ │ Markdown Chunk 3 [vec3] │ └────────────────────────────────────────────────────────────┘ ┌────────────────────────────────────────────────────────────┐ │ RETRIEVAL (在线 / 每次查询时) │ │ │ │ 用户问题 ──→ Embedder ──→ 向量搜索 ──→ Top-K chunks │ │ [query_vec] Vector DB ["文档1","文档3"...]│ └────────────────────────────────────────────────────────────┘ ┌────────────────────────────────────────────────────────────┐ │ GENERATION (增强生成) │ │ │ │ System: "基于以下文档回答用户问题..." │ │ Context: [文档1内容, 文档3内容, ...] │ │ User: 用户原始问题 │ │ ──→ LLM ──→ 基于真实文档的准确回答 ✅ │ └────────────────────────────────────────────────────────────┘

3.2 文本切分策略

3.3 完整 RAG 实现

class RAGPipeline:
    """完整的 RAG Pipeline —— Index + Retrieve + Generate"""

    def __init__(self, embedder, vector_db, llm):
        self.embedder = embedder
        self.db = vector_db
        self.llm = llm

    async def index(self, documents: list[str], metadatas: list[dict] = None):
        """离线：索引文档"""
        for doc, meta in zip(documents, metadatas or [{}]*len(documents)):
            chunks = splitter.split_text(doc)
            embeddings = self.embedder.embed_batch(chunks)
            self.db.insert(chunks, embeddings, meta)

    async def query(self, question: str, top_k: int = 5) -> str:
        """在线：检索 + 生成"""

        # Step 1: 向量检索
        query_vec = self.embedder.embed(question)
        retrieved = self.db.search(query_vec, k=top_k)

        # Step 2: 构造增强 prompt
        context = "\n\n---\n\n".join(r["text"] for r in retrieved)
        system = f"""You are a knowledgeable assistant. Answer the user's question
based ONLY on the provided context. If the context doesn't contain
the information, say "I don't have enough information to answer."

Context:
{context}"""

        # Step 3: LLM 生成
        response = await self.llm.chat(messages=[
            {"role": "system", "content": system},
            {"role": "user", "content": question},
        ])

        return response.content

3.4 RAG 高级技巧

技巧原理效果
HyDE (假设文档嵌入)让 LLM 先生成一份假答案 → 用假答案 embedding 去检索召回率 +10-20%
Re-ranking第一步检索 20 条 → 第二步用 Cross-Encoder 精确排 top-5精度大幅提升
混合检索 (Hybrid)向量检索 (语义) + BM25 (关键词) 加权合并兼顾语义+精确匹配
上下文压缩检索后让 LLM 从 chunk 中只提取与问题相关的句子减少无关噪音
Multi-Query用 LLM 把一个问题改写为 3-5 个变体 → 分别检索 → 去重合并覆盖多种表述

4Agent 记忆系统

RAG 解决外部知识，但 Agent 还需要自身的记忆——记住用户偏好、记住会话历史、记住学到的教训。

4.1 三层记忆架构

┌─────────────────────────────────────────────────────────┐ │ Layer 1: Working Memory (工作记忆) │ │ 当前会话的 messages 数组 (in-context) │ │ 容量: 受 context window 限制 │ │ 生命周期: 单次对话 │ │ "用户刚才说想学 Python" │ ├─────────────────────────────────────────────────────────┤ │ Layer 2: Short-Term Memory (短期记忆) │ │ 近期的会话摘要、关键决策 (存入向量库) │ │ 容量: 几千条 │ │ 生命周期: 几天到几周 │ │ "上周用户提到他在做 Web 开发项目" │ ├─────────────────────────────────────────────────────────┤ │ Layer 3: Long-Term Memory (长期记忆) │ │ 用户画像、持久偏好、学到的经验 (结构化存储) │ │ 容量: 理论上无限 │ │ 生命周期: 永久 │ │ "用户偏好：简洁回复、中文交互、Mac 环境" │ └─────────────────────────────────────────────────────────┘

4.2 记忆管理器实现

class AgentMemory:
    """Agent 的三层记忆系统"""

    def __init__(self, embedder, vector_db, llm):
        self.embedder = embedder
        self.db = vector_db      # Layer 2 + Layer 3 都存在这里
        self.llm = llm
        self.working_memory = []  # Layer 1: messages list

    # === Layer 1: 会话内工作记忆 ===
    def add_to_working(self, role: str, content: str):
        self.working_memory.append({"role": role, "content": content})
        self._trim_if_needed()

    def _trim_if_needed(self):
        """工作记忆超过 token 预算时自动压缩"""
        total_tokens = count_tokens(self.working_memory)
        if total_tokens > MAX_WORKING_TOKENS:
            # 保留 system prompt + 最近 10 轮，其余用 LLM 摘要
            ...

    # === Layer 2: 短期记忆 (向量检索) ===
    async def remember_short_term(self, content: str, meta: dict):
        """把重要信息写入短期记忆"""
        embedding = self.embedder.embed(content)
        self.db.insert(
            text=content,
            vector=embedding,
            metadata={**meta, "type": "short_term", "ts": time.time()},
        )

    async def recall_short_term(self, query: str, k=5) -> list[str]:
        """检索相关的短期记忆"""
        query_vec = self.embedder.embed(query)
        results = self.db.search(query_vec, k=k)
        return [r["text"] for r in results]

    # === Layer 3: 长期记忆 (结构化持久化) ===
    async def persist_long_term(self, user_id: str, fact: str):
        """持久化长期事实（用户画像、偏好等）"""
        self.db.insert(text=fact, vector=None,
                       metadata={"type": "long_term", "user_id": user_id})

4.3 记忆的生命周期管理

操作触发时机实现
写入每轮对话结束 / 用户说"记住这个" / LLM 判断重要异步写入向量库
检索每次新请求开始时，用当前问题检索相关记忆向量搜索 + 注入 system prompt
遗忘短期记忆过期 (TTL)；用户要求删除按时间戳过滤 + metadata 标记
合并发现重复/矛盾记忆时LLM 判断 → 去重 → 合并为新版本
升级短期记忆被反复检索 → 升级为长期记忆访问频率加权 + 阈值触发

4.4 记忆注入 Agent Context

async def build_agent_context(memory: AgentMemory, user_input: str) -> list:
    """构建 Agent 的完整 context（融合三层记忆）"""

    messages = []

    # 1. System Prompt
    system = "You are a helpful assistant."

    # 2. 注入长期记忆（用户画像）
    long_term = memory.db.search("user profile", filter={"type": "long_term"})
    if long_term:
        system += "\n\n## User Profile\n" + "\n".join(f"- {m}" for m in long_term)
    messages.append({"role": "system", "content": system})

    # 3. 检索相关短期记忆
    recalled = await memory.recall_short_term(user_input, k=3)
    if recalled:
        messages.append({"role": "system", "content":
            f"## Related Memories\n" + "\n".join(f"- {r}" for r in recalled)})

    # 4. 工作记忆（最近对话）
    messages.extend(memory.working_memory)

    # 5. 当前用户输入
    messages.append({"role": "user", "content": user_input})

    return messages

5多模态检索

Agent 不仅要检索文本，还要能「看懂」图片、搜索视频、理解音频。多模态 RAG 是下一代 Agent 的标配。

5.1 多模态 Embedding

模型支持模态维度特点
CLIP (OpenAI)文本 + 图像512/768开源标杆，文本搜图/图搜图
SigLIP文本 + 图像768比 CLIP 更好
ImageBind (Meta)图 + 文 + 音 + 深度 + 热力 + IMU1024六模态统一空间
jina-clip-v2文本 + 图像102489B 训练数据，多语言

5.2 CLIP 实战：以文搜图

from transformers import CLIPProcessor, CLIPModel
from PIL import Image

model = CLIPModel.from_pretrained("openai/clip-vit-base-patch32")
processor = CLIPProcessor.from_pretrained("openai/clip-vit-base-patch32")

# 索引：图像 → 向量
def encode_image(image_path: str) -> list[float]:
    img = Image.open(image_path)
    inputs = processor(images=img, return_tensors="pt")
    with torch.no_grad():
        emb = model.get_image_features(**inputs)
    return emb[0].numpy().tolist()

# 搜索：文本 → 向量 → 找最相似图片
def encode_text(text: str) -> list[float]:
    inputs = processor(text=[text], return_tensors="pt")
    with torch.no_grad():
        emb = model.get_text_features(**inputs)
    return emb[0].numpy().tolist()

5.3 Agent 中的多模态 RAG 流程

用户: "上次发给我的那个架构图里，数据库在哪个位置？" Step 1: 文本 Embedding → 在图像描述向量库中找到"架构图" Step 2: 取出图像 → 发给 Vision LLM (gpt-4o / Claude Vision) Step 3: Vision LLM: "数据库位于图的右下角，标注为 'PostgreSQL Cluster'" Step 4: 返回给用户: "数据库在你上次的架构图的右下角，标注为 PostgreSQL Cluster"
