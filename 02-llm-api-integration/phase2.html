1 Transformer 原理概览

不需要精通训练细节，但必须理解「为什么 LLM 会这样回答」——注意力机制、上下文窗口、Tokenization、自回归生成。这些决定了 Agent 后端的设计上限。

1.1 从 RNN 到 Transformer —— 为什么是革命？

RNN/LSTM (旧范式): "今天" → RNN → "天气" → RNN → "真" → RNN → "好" ↑ 必须串行！无法并行训练。token 距离远了会遗忘。 Transformer (2017, "Attention Is All You Need"): ┌─────────────────────────────┐ "今天" ──→│ │──→ 0.1 "天气" ──→│ Self-Attention │──→ 0.3 "真" ──→│ (每个 token 同时看所有 token)│──→ 0.5 "好" ──→│ │──→ 0.1 └─────────────────────────────┘ ↑ 完全并行！任意距离的 token 关系都能一次计算。

1.2 核心组件（足够理解 LLM 行为即可）

组件作用Agent 意义
Self-Attention每个 token 计算与所有其他 token 的关联权重决定了模型「注意到」上下文的哪部分。Prompt 中关键信息要放在注意力最强的位置（开头和结尾）
Multi-Head Attention多组并行的注意力（不同「视角」）解释模型为何能同时关注语义、语法、指代
Positional Encoding给 token 注入位置信息顺序很重要！不要随意打乱 system/user/assistant 消息顺序
LayerNorm + Residual稳定深层网络训练—
Feed-Forward (FFN)每个 token 独立的非线性变换（知识存储层）模型知识的「记忆」主要存在这里
Causal Mask防止看到未来 token（三角形掩码）这就是为什么 LLM 是自回归的一一只能从左到右生成

1.3 Tokenization —— 被低估的核心

LLM 处理的是 token，不是字符。Tokenization 把文本切成 token ID 序列。同一个词在不同 tokenizer 下可能切成不同数量的 token。

# 用 tiktoken (OpenAI 的 tokenizer) 演示
import tiktoken

enc = tiktoken.encoding_for_model("gpt-4")
tokens = enc.encode("Hello, 你好世界！")
print(tokens)  # [9906, 11, 57668, 53901, 3574, 244, 98220, 6447]
print(len(tokens)) # 8 — "Hello"=1 token, "你好世界"=好几个 token

# 中英文 token 效率差异巨大：
english = enc.encode("artificial intelligence")  # 2 tokens
chinese = enc.encode("人工智能")                   # 3 tokens ← 同样语义，多 50%

# 各模型 tokenizer 不同：
# GPT-4    → cl100k_base (tiktoken)
# Claude   → 不公开 tokenizer，靠 API 返回 usage 统计
# DeepSeek → 自己的 BPE tokenizer
# Llama    → SentencePiece (BPE)
Agent 实战教训：Token 数量 ≠ 字符数量。中文内容是 Token 大户（平均 1 个汉字 = 1.5~2 token）。在做 Token 预算管理时一定要用对应的 tokenizer 精确计算，不能用「字数 × 2」这种粗略估算。

1.4 自回归生成 (Autoregressive Generation)

LLM 生成是走一步看一步的：每次输入全部历史，输出下一个 token，然后把这个新 token 拼回去，继续生成。这就是为什么 context 越长越慢越贵 —— 每次都要把整个上下文重新编码。

输入: [system: "你是助手"] [user: "1+1=?"] ↓ 前向传播 输出: "2" ↓ 拼回去 输入: [system: "你是助手"] [user: "1+1=?"] [assistant: "2"] ↓ 再次前向传播（整个序列重算！） 输出: "。" ↓ 拼回去 输入: [system: "你是助手"] [user: "1+1=?"] [assistant: "2。"] ↓ ... 输出: 结束信号 (EOS / stop token) 这就是 Chat Completions API 的底层： messages 数组每次递增，模型重复计算整个序列。

1.5 关键参数 —— Agent 必须掌握

参数含义推荐范围Agent 用途
temperature随机度，越高越「发散」，越低越「保守」0~2创造类任务 0.8~1.2；事实/代码/calling 0~0.3
top_p核采样——只从概率累积到 top_p 的 token 中选0~1与 temperature 二选一（不要同时调）
max_tokens输出 token 上限（不含输入）视需求防止无限生成烧钱；工具调用设小点
stop停止序列（最多 4 个）如 ["\n\n"]让 LLM 提前结束，控制输出格式
presence_penalty降低已出现 token 概率（-2~2）减少重复
frequency_penalty按频率降低 token 概率（-2~2）按频率降低 token 概率（-2~2）减少高频词，增加多样性
seed随机种子（确定性输出）任意整数测试、调试、回归验证
temperature=0 的陷阱： 不会完全确定性。浮点运算的微小差异仍会产生不同结果。要真正确定性，同时设 temperature=0 + seed=N（OpenAI 支持，但不是所有 Provider 都支持）。

1.7 预填充 (Prefill) vs 解码 (Decode) —— 理解成本结构

阶段计算方式速度成本
Prefill
(处理输入)		并行处理所有输入 token
一次前向传播算完整个 prompt		很快按 input token 计费
(便宜)
Decode
(生成输出)		逐 token 自回归生成
每次都要重算整个序列		慢按 output token 计费
(贵 2~10×)
省钱原则：短 prompt 让 LLM 生成长回复 = 小 input × 大 output = 贵！
长 prompt 让 LLM 给简短答案 = 大 input × 小 output = 在可控范围。
Agent 设计的核心权衡：尽量把信息放 prompt 里（便宜），别让 LLM 自己去「想」（贵）。

import openai

client = openai.OpenAI(api_key="sk-...")

response = client.chat.completions.create(
    model="gpt-4o",
    messages=[
        {"role": "system",  "content": "You are a coding assistant."},
        {"role": "user",    "content": "Write a hello.py"},
        # role 可以是: system, user, assistant, tool, function(deprecated)
        # content 可以是: str 或 [{"type":"text","text":"..."}, {"type":"image_url",...}]
    ],
    temperature=0.3,
    max_tokens=2000,
    top_p=1.0,
    n=1,                          # 生成几个候选（>1 时 choices 数组变长）
    stop=["\n\n\n"],                 # 触到这些字符串立即停止
    presence_penalty=0,
    frequency_penalty=0,
    seed=42,                       # 确定性（需同时 temperature=0）

    # ---- 流式 ----
    stream=False,                   # True → 返回 SSE 迭代器
    stream_options={"include_usage": True},  # 流式时也能拿 usage

    # ---- 函数调用（详见阶段3）----
    tools=[...],
    tool_choice="auto",

    # ---- 结构化输出 ----
    response_format={"type": "json_object"},  # 或 json_schema

    # ---- Logprobs ----
    logprobs=True,
    top_logprobs=5,               # 返回 top-5 备选 token 及概率
)

# 响应结构
msg = response.choices[0].message
print(msg.content)                       # "Here's a hello.py:\n\nprint('Hello')"
print(msg.role)                          # "assistant"
print(msg.tool_calls)                    # None (没触发 function calling)
print(response.usage)                    # CompletionUsage(prompt_tokens=32,

print(response.model)                    # "gpt-4o-2024-05-13"
print(response.system_fingerprint)       # "fp_xxx" ← 模型版本指纹
print(response.id)                       # "chatcmpl-xxx" ← 唯一请求 ID

2.5 Logprobs —— 调试神器

Logprobs 告诉你模型在每个位置「想说什么但又没说」。在 Agent 场景下用于：路由准确性检测、幻觉概率估计、结构化输出校验。

response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "1+1=?"}],
    logprobs=True,
    top_logprobs=3,
)

choice = response.choices[0]
for token_info in choice.logprobs.content:
    print(f"Token: {token_info.token}")
    print(f"  Prob: {round(exp(token_info.logprob)*100, 1)}%")
    for alt in token_info.top_logprobs:
        print(f"  alt: {alt.token} ({round(exp(alt.logprob)*100, 1)}%)")

# 输出:
# Token: 2
#   Prob: 99.8%
#   alt:  (0.1%)
#   alt: 1 (0.05%)
#   alt:   (0.03%)

# Agent 实战：用 logprobs 检测「模型是否真的确定了」
def is_model_sure(logprob_content, threshold=0.9):
    """如果模型对每个 token 都 > 90% 确定，说明输出可信"""
    return all(
        exp(t.logprob) > threshold for t in logprob_content
    )

3.2 统一抽象接口设计

from abc import ABC, abstractmethod
from dataclasses import dataclass, field
from typing import AsyncIterator, Optional

@dataclass
class LLMMessage:
    role: str                # system | user | assistant | tool
    content: str
    tool_call_id: Optional[str] = None
    tool_calls: Optional[list] = None

@dataclass
class LLMResponse:
    content: str
    tool_calls: list = field(default_factory=list)
    usage: dict = field(default_factory=dict)
    model: str = ""
    finish_reason: str = ""

@dataclass
class LLMStreamChunk:
    delta_content: Optional[str] = None
    delta_tool_calls: list = field(default_factory=list)
    finish_reason: Optional[str] = None
    usage: Optional[dict] = None

class BaseLLMProvider(ABC):
    """所有 LLM Provider 必须实现的统一接口"""

    @abstractmethod
    async def chat(self,
              messages: list[LLMMessage],
              temperature: float = 0.7,
              max_tokens: int = 4096,
              tools: Optional[list] = None) -> LLMResponse:
        """非流式调用"""
        ...

    @abstractmethod
    async def chat_stream(self,
                   messages: list[LLMMessage],
                   temperature: float = 0.7,
                   max_tokens: int = 4096,
                   tools: Optional[list] = None) -> AsyncIterator[LLMStreamChunk]:
        """流式调用"""
        ...

    @property
    @abstractmethod
    def model_name(self) -> str: ...

    @abstractmethod
    def count_tokens(self, text: str) -> int: ...

3.3 OpenAI Provider 实现

class OpenAIProvider(BaseLLMProvider):
    def __init__(self, api_key: str, model: str = "gpt-4o-mini"):
        self._model = model
        self.client = openai.AsyncOpenAI(api_key=api_key)

    async def chat(self, messages, temperature=0.7, max_tokens=4096, tools=None):
        resp = await self.client.chat.completions.create(
            model=self._model,
            messages=[{"role": m.role, "content": m.content} for m in messages],
            temperature=temperature, max_tokens=max_tokens,
            tools=tools,
        )
        choice = resp.choices[0]
        return LLMResponse(
            content=choice.message.content or "",
            usage={"input": resp.usage.prompt_tokens,
                   "output": resp.usage.completion_tokens},
            model=resp.model,
            finish_reason=choice.finish_reason or "",
        )

    @property
    def model_name(self): return self._model

    def count_tokens(self, text):
        return len(tiktoken.encoding_for_model(self._model).encode(text))

3.4 Anthropic Provider —— 处理差异

class AnthropicProvider(BaseLLMProvider):
    def __init__(self, api_key: str, model: str = "claude-3-5-sonnet-20241022"):
        self._model = model
        self._key = api_key
        self._url = "https://api.anthropic.com/v1/messages"

    def _convert_messages(self, messages: list[LLMMessage]):
        """Anthropic 的消息格式完全不同！"""
        system_prompt = ""
        anthropic_msgs = []
        for m in messages:
            if m.role == "system":
                system_prompt = m.content  # Anthropic 的 system 是顶层参数!
            else:
                anthropic_msgs.append({
                    "role": m.role,
                    "content": m.content,
                })
        return system_prompt, anthropic_msgs

    async def chat(self, messages, temperature=0.7, max_tokens=4096, tools=None):
        system, an_msgs = self._convert_messages(messages)
        headers = {
            "x-api-key": self._key,
            "anthropic-version": "2023-06-01",
            "content-type": "application/json",
        }
        body = {
            "model": self._model,
            "max_tokens": max_tokens,
            "messages": an_msgs,
            "temperature": temperature,
        }
        if system:
            body["system"] = system       # ← 关键差异!
        if tools:
            body["tools"] = tools          # Anthropic tool 格式也不同

        async with aiohttp.ClientSession() as session:
            async with session.post(self._url, json=body, headers=headers) as resp:
                data = await resp.json()
                return LLMResponse(
                    content=data["content"][0]["text"],
                    usage={"input": data["usage"]["input_tokens"],
                           "output": data["usage"]["output_tokens"]},
                    model=data["model"],
                    finish_reason=data["stop_reason"] or "",
                )

3.5 Provider 工厂 + 动态路由

class ProviderRouter:
    """根据模型名、任务类型、预算自动选择 Provider"""

    def __init__(self, providers: dict[str, BaseLLMProvider]):
        self.providers = providers

    def get(self, model: str) -> BaseLLMProvider:
        key = model.split("/")[0]  # "openai/gpt-4o-mini" → "openai"
        if key not in self.providers:
            raise ValueError(f"Unknown provider: {key}")
        return self.providers[key]

    def route_by_task(self, task_type: str) -> BaseLLMProvider:
        """简单路由：代码→Claude, 对话→DeepSeek, 视觉→OpenAI"""
        routing = {
            "code": "anthropic",
            "chat": "deepseek",
            "vision": "openai",
        }
        return self.providers[routing.get(task_type, "openai")]
为什么不用 LiteLLM / LangChain？
它们确实封装好了多 Provider，但：① 依赖重、版本冲突多；② 抽象层太厚，出 bug 难调试；③ Agent 需要精细控制（retry、circuit breaker、自定义路由），自己写 200 行就够了。理解底层后，要不要用框架是你的选择。

4 Prompt Engineering

Agent 的 prompt 不是写死的文案，而是动态构建的控制程序。你的 prompt 能力直接决定 Agent 的可靠性。

4.1 Prompt 的四个层次

Level 1: System Prompt (角色 + 规则) → "You are a coding assistant. Always use TypeScript. Never apologize." Level 2: Context Injection (动态上下文) → "Current date: 2025-06-05\nUser's OS: Windows 10\nGit branch: main" Level 3: Tools Description (工具声明) → functions: [{name: "search", description: "搜索知识库", parameters: {...}}] Level 4: Conversation History (对话历史) → [user: "Hello", assistant: "Hi!", user: "帮我写代码", ...] 这四个层次 从上到下越来越动态。System Prompt 可能几周不变， Conversation History 每次请求都变。总 token 数必须控制在上下文窗口内。

4.2 System Prompt 设计原则

# ❌ 糟糕的 system prompt —— 模糊、矛盾、遗漏
"You are a helpful AI. Be nice and answer questions."

# ✅ 好的 system prompt —— 清晰、具体、可测试
"""
You are a terminal-based coding assistant. Follow these rules:

1. LANGUAGE: Always write Python 3.11+ code unless asked otherwise.
2. PACKAGES: Only use stdlib + 'httpx', 'pydantic'. Do NOT use requests.
3. FORMAT: Return ONLY the code block, no explanation before or after.
4. ERROR HANDLING: Always include try/except with specific exceptions.
5. TERMINATION: If user asks for a plan, output as markdown. No code.
6. CONSTRAINT: Never create files larger than 500 lines.
7. OUTPUT: Print results to stdout, not files, unless asked.
"""

Agent System Prompt 检查清单：

  • ✅ 有没有明确定义角色？（"You are X"）
  • ✅ 有没有说明约束？（"Never do X", "Always do Y"）
  • ✅ 有没有给正反例？（"Good: ... Bad: ..."）
  • ✅ 有没有定义输出格式？（"Return as JSON"）
  • ✅ 有没有错误的兜底策略？（"If unsure, say I don't know"）
  • ✅ 有没有加上工具使用指引？（"Use tool X for Y"）
  • ❌ 不要写自相矛盾的规则（"Be concise" + "Give detailed explanations"）
  • ❌ 不要写无法执行的要求（"Always be correct"）

5 Token 计算与成本优化

Agent 跑起来后，Token 成本是最大的运营开销。精确计算、精细控制、不断优化。

5.1 Token 预算模型

5.2 精确 Token 计数

import tiktoken

class TokenCounter:
    """支持多 Provider 的精确 token 计数"""

    ENCODINGS = {
        "gpt-4o": "o200k_base",
        "gpt-4": "cl100k_base",
        "gpt-3.5-turbo": "cl100k_base",
    }

    def __init__(self, model: str = "gpt-4o-mini"):
        enc_name = self.ENCODINGS.get(model, "cl100k_base")
        self.enc = tiktoken.get_encoding(enc_name)
        self.model = model

    def count(self, text: str) -> int:
        return len(self.enc.encode(text))

    def count_messages(self, messages: list) -> int:
        """仿 OpenAI 的 token 计算（含 3 token/message 开销）"""
        total = 0
        for msg in messages:
            total += 3  # message overhead
            for key, value in msg.items():
                total += self.count(str(value))
        total += 3  # reply priming
        return total

    def will_fit(self, messages: list, max_output: int,
               context_window: int = 128000) -> bool:
        return self.count_messages(messages) + max_output <= context_window

5.3 滑动窗口裁剪 —— Agent 长会话的必备技能

class SlidingWindowTrimmer:
    """在 token 预算内保留最重要的消息"""

    def __init__(self, max_input_tokens=6000):
        self.max_input = max_input_tokens
        self.counter = TokenCounter()

    def trim(self, messages: list) -> list:
        """保留 system + 最后 N 条消息，中间切掉"""
        if self.counter.count_messages(messages) <= self.max_input:
            return messages

        result = []
        # 1. System prompt 永远保留（通常是 messages[0]）
        system_msgs = [m for m in messages if m["role"] == "system"]
        result.extend(system_msgs)

        # 2. 从后往前加非 system 消息，直到 token 超限
        remaining = messages[len(system_msgs):]
        tail = []
        for msg in reversed(remaining):
            candidate = [msg] + tail + result
            if self.counter.count_messages(candidate) <= self.max_input:
                tail.insert(0, msg)
            else:
                break

        return result + tail

5.4 成本追踪系统

from datetime import datetime
import json

class CostTracker:
    """每次 LLM 调用的成本记录"""

    PRICING = {  # $/1M tokens
        "gpt-4o": (2.50, 10.00),
        "gpt-4o-mini": (0.15, 0.60),
    }

    def __init__(self):
        self.records = []

    def record(self, model: str, input_tokens: int, output_tokens: int):
        in_price, out_price = self.PRICING.get(
            model.split("-20")[0], (0, 0)
        )
        cost = (input_tokens * in_price + output_tokens * out_price) / 1_000_000
        self.records.append({
            "time": datetime.now().isoformat(),
            "model": model,
            "input_tokens": input_tokens,
            "output_tokens": output_tokens,
            "cost_usd": round(cost, 6),
        })

    def summary(self) -> dict:
        total_cost = sum(r["cost_usd"] for r in self.records)
        total_calls = len(self.records)
        return {
            "total_calls": total_calls,
            "total_cost": round(total_cost, 4),
            "avg_cost_per_call": round(total_cost / total_calls, 6) if total_calls else 0,
        }

6 结构化输出

Agent 不能依赖 LLM 的「自由发挥」。让 LLM 返回稳定的结构化数据，是所有 downstream logic（工具调用、状态机、多 Agent 通信）的根基。

6.1 四种结构化输出方案对比

6.2 OpenAI Structured Outputs (JSON Schema) —— 最新最可靠

from pydantic import BaseModel
from typing import Literal, Optional

class CodeReviewResult(BaseModel):
    """用 Pydantic 定义输出结构 → 自动生成 JSON Schema"""
    status: Literal["pass", "fail", "needs_improvement"]
    summary: str
    issues: list[str] = []
    score: int  # 0-100

    class Severity(BaseModel):
        critical: int = 0
        major: int = 0
        minor: int = 0

    severity: Severity

response = client.chat.completions.create(
    model="gpt-4o-2024-08-06",  # ★ 支持 Structured Outputs 的模型
    messages=[{"role": "system", "content": "Review code"},
              {"role": "user", "content": code}],
    response_format={
        "type": "json_schema",
        "json_schema": {
            "name": "code_review",
            "schema": CodeReviewResult.model_json_schema(),
            "strict": True,  # ★ 100% 符合 schema，否则报错
        }
    }
)

result = CodeReviewResult.model_validate_json(

)

print(result.score)    # 85
Structured Outputs 限制： ① 只有部分模型支持（gpt-4o-2024-08-06 及以后版本）；② strict=true 时不允许 optional 字段、不允许 additionalProperties；③ 不支持 minItems/maxItems 约束（OpenAI 可能在后续版本添加）。

6.3 Tool Calling 作结构化输出

Tool Calling 本质是「让 LLM 以函数参数形式返回结构化数据」。对于不支持 JSON Schema 的模型（如 DeepSeek），这就是最好的结构化方案。

response = client.chat.completions.create(
    model="deepseek-chat",
    messages=[{"role": "user", "content": "Review this code:\n\n" + code}],
    tools=[{
        "type": "function",
        "function": {
            "name": "submit_review",
            "description": "Submit a code review",
            "parameters": {
                "type": "object",
                "properties": {
                    "status": {"type": "string", "enum": ["pass", "fail"]},
                    "score": {"type": "integer", "minimum": 0, "maximum": 100},
                    "issues": {"type": "array", "items": {"type": "string"}},
                },
                "required": ["status", "score"],
            }
        }
    }],
    tool_choice={"type": "function", "function": {"name": "submit_review"}},
)

tool_call = response.choices[0].message.tool_calls[0]
review_data = json.loads(tool_call.function.arguments)
print(review_data)  # {"status": "pass", "score": 85, "issues": []}

6.4 JSON Mode 陷阱与解决方案

# ❌ 陷阱: JSON 被 markdown 代码块包裹
# LLM 返回: ```json\n{"status":"ok"}\n``` ← 不是纯 JSON！

def safe_json_parse(raw: str) -> dict:
    """鲁棒的 JSON 解析 —— 应对各种 LLM 输出"""
    raw = raw.strip()

    # 1. 去掉 markdown 代码块
    if raw.startswith("```"):
        raw = raw.split("\n", 1)[1]  # 去掉第一行 ```json
        if raw.endswith("```"):
            raw = raw[:-3]

    # 2. 提取第一个 JSON 对象/数组
    import re
    match = re.search(r'(\{.*\}|\[.*\])', raw, re.DOTALL)
    if match:
        raw = match.group(1)

    # 3. 修复常见错误
    raw = re.sub(r',\s*}', '}',  raw)  # trailing comma
    raw = re.sub(r',\s*]', ']',  raw)  # trailing comma in array

    return json.loads(raw)
最佳实践： OpenAI → 优先用 response_format={"type":"json_schema"} (strict=true)；
DeepSeek / 其他 → 用 tool_choice 强制 tool call；
不支持 tool calling 的模型 → response_format={"type":"json_object"} + prompt 指定 schema + 后端 safe_json_parse。
