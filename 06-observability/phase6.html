阶段 6：可观测性 & 评估

分布式追踪、结构化日志、Agent Eval、成本优化 —— 让 Agent 从「能跑」到「可靠」

⏱ 预计 1-2 周 | 📚 4 个核心主题

学习路径逻辑：Agent 跑起来了，但你怎么知道它在干什么？哪里慢了？为什么这次回答错了？本阶段解决「看见 Agent 内部」和「衡量 Agent 质量」的问题。追踪让你看到每一次 LLM 调用和工具执行 → 日志让你事后排查问题 → 评估让你量化 Agent 的好坏 → 成本监控让你不破产。

1分布式追踪

Agent 执行一次任务涉及多次 LLM 调用 + 多次工具执行 + 多次 RAG 检索。没有追踪 = 盲飞。

OpenTelemetry + LangFuse 集成

用户请求 (trace: abc123) ├── Span: Agent Loop Start (0ms) ├── Span: LLM Call #1 (5ms - 1200ms) │ ├── model: gpt-4o │ ├── input_tokens: 452 │ ├── output_tokens: 28 │ └── tool_calls: [search] ├── Span: Tool Execution: search (1200ms - 1500ms) │ ├── args: {query: "Python async"} │ └── result: "Found 5 docs" ├── Span: LLM Call #2 (1500ms - 2800ms) │ ├── model: gpt-4o │ └── finish_reason: stop └── Span: Agent Loop End (2800ms)
# LangFuse 集成 (OpenTelemetry 兼容)
from langfuse import Langfuse
langfuse = Langfuse(public_key="pk-...", secret_key="sk-...")

trace = langfuse.trace(name="agent-run", user_id="user_123")

async def traced_llm_call(trace, step, messages, tools):
    span = trace.span(name=f"llm-call-{step}", input={"messages": messages})
    response = await llm.chat(messages, tools)
    span.end(output={
        "content": response.content,
        "usage": response.usage,
        "model": response.model,
    })
    return response

2结构化日志

print() 不是日志系统。生产环境需要结构化、可查询、带上下文的日志。

import structlog, uuid

logger = structlog.get_logger()

async def agent_step(session_id, step_num, action):
    logger.info("agent.step",
        session_id=session_id,
        step=step_num,
        action=action,
        timestamp=time.time(),
    )

# 禁止打印完整 prompt / API Key!
def safe_log_messages(messages):
    return [{"role": m["role"], "length": len(m["content"])} for m in messages]

日志级别选择

级别Agent 场景
INFO正常步骤、工具调用、对话轮次
WARN重试、降级、截断、Token 接近上限
ERROR工具失败、LLM API 错误、超时
DEBUG完整 prompt（脱敏后）、工具参数详情

3Agent 评估体系

Agent 的「好坏」不是感觉出来的。需要系统性评估：准确性、效率、安全性、用户体验。

Eval 四维度

维度指标测量方法
任务成功率% of tasks completed correctly人工标注 + LLM-as-Judge
工具选择准确性% of correct tool selection对比 ground-truth tool 序列
效率Average steps per task / tokens per task追踪数据统计
安全性% of unsafe actions blocked红队测试 + 自动化扫描

4成本监控优化

Agent 跑一个月后，最大的惊喜（或惊吓）是 LLM API 账单。成本监控不是可选项。

成本监控仪表盘指标

指标含义告警阈值
每日总成本每天所有 LLM 调用总花费> $10/天
每次对话平均成本单次用户对话的平均花费> $0.05/对话
平均 Steps/任务Agent 完成一个任务的平均步数> 10 steps
浪费率因中断/重试/超时产生的无效 token 占比> 15%
省钱铁律：① gpt-4o-mini 做简单任务（分类/路由/摘要），gpt-4o 只做最难的部分 ② 缓存 system prompt (prompt caching 省 50% input 成本) ③ 监控 max_steps 超限率——如果 Agent 频繁跑满 max_steps，说明任务太复杂需要拆分。
